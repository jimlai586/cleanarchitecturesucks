Storyboard was a myth

Back when I started working as iOS developer, I was told that storyboard is horrible and not suitable for team collaboration.

OK. It did seem plausible. So what is the alternative? Auto-layout codes?

The team ended up using manual frame calculation.

It was a bug-ridden mess. Calculating frame manually for every component is tedious in itself, and is only worsen by the fact that there was no MVC structure to speak of and no refactoring was ever bothered. Everywhere you looked, there is a line to add subview or set border width, and you often need to scroll through tens of pages of UI codes just to inspect some control logic.

Well, that sounds like your team sucks, not that the frame calculation sucks although using auto-layout might be more up-to-date. You may respond.

Yes, that was my thought too, and I began fixing all the architectural problems when that project finished. I was preparing for an auto-layout based development and I noticed the storyboard, sitting in the dark alone.

Why bother using Xcode if not for Interface Builder

Xcode, love it or hate it?

It is surprising to see that many of my colleagues still use the same mindset and work-flow today as if today were in 2010. Barely any third-party framework, except AFNetworking for the sole purpose of detecting network reachability. Everything including UI is written in viewDidLoad(). Frame is calculated by hand and is placed in viewDidLoad() too. Such event took place in 2016. It was a sight to see.

At first I thought it might be a time-tested tradition that was proven solid again and again through project after project. This is the canonical way to do iOS, I’m just too noob to understand it; The tale told a story of the storyboard eating junior developer, and auto-layout stealing your children at night.

But frame calculation just didn’t make any sense! Any change in the frame of one component, you have to re-calcuate all others affected and it ripples out. It is hard to imagine a senior developer, say 3 ~ 5 years of experience under his belt, would still have to calculate every button and every label. That doesn’t look like a promising career path.

Since interface builder was not used anywhere, and Objective-C was the development language. I might as well use Appcode instead of Xcode.

It *clicks*. Appcode just works. It was then that I realized how anti-human Xcode’s UI design was. I want open-in-new-tab to be default when I navigate through files or when I’m debugging via breakpoints. But Xcode in its infinite wisdom denies me even that.

At this point, I thought I found the solution. Appcode + auto-layout. Time tested again and again through project after project. I spent considerable efforts to introduce auto-layout and appropriate third-party frameworks to the team, and completed the necessary footwork to set everything up and documents and examples.

It might have been a happily-ever-after and the story ends, if not for one casual google attempt.

The revenge of Storyboard

A sith lord?

It was an attempt to introduce MVC to other team members. I did not think writing-everything-in-viewdidload was in anyway an established design pattern.

Sure that is why one makes a custom view. But in the process I also noticed that it’s tiresome that one must carefully and repeatedly run the app and navigate to the right page to check if UI is correct.

OK, after some googling, found IBDesignable and XIB to give a preview of auto-layout codes. But XIB did not support prototype cell, which prevents preview of tableview, which is like 70% of the views.

OK, so drop XIB, and use storyboard. No wait. If I’m going to use storyboard anyway, why bother writing auto-layout codes first and review in storyboard later with all the overheads of IBDesignable; I might as well just draw it on storyboard.

I then googled some classic discussion on code vs xib vs storyboard. The result was … inconclusive. A problem was that many articles were dated, and might not be accurate after years of updates. One in particular caught my eyes, it was written a couple years ago and said that he had a succcess in using storyboards in a small team setup.

Well, if some guy could pull this off that many years ago, there is no reason I couldn’t pull this off now given that I’m also in a small team.

At first, it was as frustrating as I remembered when I started learning iOS. But there is a difference between things being difficult and things being unfamiliar. Maybe it was because Xcode updates, or I just git gud, everything started to make sense. It eventually *clicks*.

It is not that hard, if you avoid this, and do that when that other thing happens; oh and remember to do this after that, you will see that it just works.

It sounds like sarcasm. But it is not. Because the pay off is more than worth the investment, at least for my projects. I can understand that why storyboard is a no-go in early days, or even auto-layout in earlier days, but at the time I’m writing this, which is less than a month away from 2018, I don’t see any problems using it in team collaboration, and I am using it in team collaboration.

I think 80–20 principle applies well in Cocoa-touch. You do 80% of routine view setup in a custom view or storyboard, and do 20% further setup in viewDidLoad() to account for runtime situations. 80% of the views are just run-of-the-mill that storyboard is more than capable of handling; 20% seems a reasonable ratio that a developer should pay more attention to get it working. If you are working for a project that requires 100% special treatment for views, then by all means don’t use it. But again 80% of developers might not be in that position.

On the other hand, developer skill level is another factor. My colleagues would write the code as is, i.e.; border width, border color, font size, text color, addSubview… etc; and repeat everything for next component. So when you review a custom view, you would see corner radius is set every 10 lines, and no one bothers to create a RoundButton and go from there.

Is storyboard stronger?

No, no, no. Quicker, easier, more seductive.

Thank you master Yoda. I’ll take quicker and easier.

In current project, which consists of 3 ~ 4 people, I did 80% up of UI and UI-related API. This number of course means nothing without further context.

But what I can tell you is that, if you plan on doing UI codes, be sure you can do it at minimum roughly the same speed as that of storyboard. Because it is fast. You don’t see Apple promoting you to write more UI code and help you do that in coming Xcode releases. Apple wants you to use storyboard, and put efforts in improving it.

That does not mean I would hit a wall when a view is special and cannot be dealt with storyboard. I simply change gear, write awesome code-based custom view and move on. The heart of it all is that storyboard is really fast in handling simple views, so fast I don’t bother setting it up in code. I can see result, resolve any and all auto-layout issues in design time; by using storyboard references, I can keep storyboard files to an atomic volume so there’s hardly any merge conflict; and in case it does happen, just use one of the two version without actually getting into resolving the conflict.

It is embarrassing to see a team starting out to avoid merge conflicts associated with storyboard, and see it taking double the time and ten times of UI codes, (and because it took so much time, there is no time to write tests) then hear it claiming storyboard is bad for team work.

Take all above with a grain of salt, because obviously I’m not in some elite seasoned developer team. I’m in a team that made storyboard look good.
