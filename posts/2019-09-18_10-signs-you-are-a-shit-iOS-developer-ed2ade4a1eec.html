Custom view without view

True story. This guy submits a merge request saying custom view. It contains a view controller managed by NSObject.

So you have to create this NSObject, it “automagically” creates a view controller that creates appropriate view for you. The view creation is not separate in any way, shape, or form, but is rather tightly coupled with state changes of view controller.

OK. Then this must be some godly dynamic view generation with complex states and user interactions.

No. It’s a menu of UI buttons whose width is dependent on another view controller. I can do this with just Storyboard in my sleep.

Not to mention, what is this concept of view controller controlled by NSObject? iOS cannot manage view controller hierarchy for you with this NSObject in the way, so you have to manage yourself.

Oh I haven’t got to the best part. This commit is merged.

View model without view nor model

True story. This guy submits a XXXVM. A view model? OK. Then I opened it.

It contains nothing but escaping closures.

Escaping closures are control. You don’t put nothing but control in something called view model.

You may argue that it’s not control when it’s updating view. But there are better ways to do that, and usually one thing would lead to another. You may start with the intention of updating view, next thing you or your co-worker would start navigating to new view controller, mutating data without notifying.

MVVM people have this obsession of moving codes from view controller to something called view model (or presenter whatever). This does not reduce the amount of work, you still need to do whatever you do in view controller. But now you do it with extra overheads. Again, there are ways to do this without view model.

More often than not, people would submit MVVM codes that I can rewrite with less than half of lines of code. On top of that I have to point out and remove all kinds of MVC and basic principle violations.

You have to be self-aware enough to determine if you are writing refactor or boilerplate. If you write boilerplate and call it refactor or MVVM, then you are a shit developer.

This counts a 3 signs. USD stands for Uni-directional, Single Responsibility/Source of truth, Dependency Injection.

Yeah yeah, a tale as old as MVC. Yet people just don’t seem to get it.

Especially for MVVM people. For some reason rules don’t apply to them.

Single responsibility/source of truth is another common pitfall.

Say you have a var data: JSON. Your view controller need a copy of this, so does your view model. What will you do?

vc.data = data
vc.vm.data = data

Because you now need to maintain consistency between these copies of data.

(two commits later)

func onNewData(_ data: JSON) {
      self.data = data
      vm.data = data
      // data and vm are now private
}

Let’s forget that view controller may still change data internally without updating view model’s copy. Let’s not worry about additional access control.

Why two copies in the first place? (or why MVVM …)

Use godamn computed property. Maintain single source of truth, don’t f-ing bother with splitting responsibility with presenter by not using presenter.

If you write massive view controller, it’s not iOS SDK sucks. It is you.

View controller inheritance

Google why OOP sucks. Even if it doesn’t suck, people suck.

If you give people chances to mess up your code, they will take it.

Even if you don’t give people chances to mess up your code, they will inject inheritance to your view controller, and mess up your code by messing up parent view controller.

Generally speaking, view inheritance is OK (SwiftUI uses View as protocol though), but controller inheritance never ends well. The problem is of course, control. Control logic changes all the time, because UI, API changes all the time. Code itself changes all the time. When you inherit view controller, you inherit all its states, which may change all the time. Plus the fact that few people actually respect boundaries between objects, and will casually mutate parent view controller without sending proper message.

The end result is that, either you screw up and some people get f-ed or vice versa. You lose nothing without inheritance, and you work much efficiently without this dependency.

Good old MVC violations

Water is wet, and people will violate MVC. Especially MVVM people.

// in UITableViewCell
let table = superView as! UITableView
table.reloadData()

We have ourselves

  1. uni-directional violation
  2. encapsulation violation
  3. force-cast violation
  4. MVC violation, a cell can control tableview to reloadData, which in turn mutates internal states of some view controller without notifying it

Just to name a few.

If you write shit in MVC, you write shit everywhere else
