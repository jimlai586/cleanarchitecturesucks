VIPER is the pinnacle of over-engineering and here is why

Just came across this gem

How to build SwiftUI apps using VIPER? - The.Swift.Dev.
There are literally hundreds of SwiftUI tutorials around the web, but I was only able to find just one or two that…theswiftdev.com

Let me make this perfectly clear.

It should be illegal to use VIPER architecture in any production code.

The main reason is this:

Look at how much boilerplate you have to write for a TODO app.

Let me say that again, a TODO app.

And this guy had the audacity to call bullshit on people calling it bullshit to use VIPER in smaller projects.

Quoting this guy:

I believe that I’ve created a modern & relatively simple pattern that can be used for literally anything.

No you have not, you have created a boilerplate mess that can’t even do a TODO app right

Learning VIPER will definitely improve your code quality thanks to the clean architecture and the SOLID principles

There’s one principle you seem to be forgetting.

It’s called DONT-OVERENGINEER.

Let me show you what a TODO app may look like in SwiftUI

struct Todo: View {
    @State var data = [TodoItem]()
    var body: some View {// do your view}
    func addItem(_ item: TodoItem) {// mutate data}
    func removeItem(at index: Int) { // mutate data}
}

Nothing beats simplicity. Nothing.

Even at scale, I can confidently say it scales a lot better than your VIPER bullshit. You can’t even make a short and concise TODO app!

And notice that mutations are functions, which can be easily refactored out in protocol extension. And that protocol can have meaningful name like

protocol TodoMutationRules {...}

Instead of

protocol TodoPresenterRouterInterface: PresenterRouterInterface {}

Which tells you jack-shit about business logic. It’s a meta-protocol. And it’s not just one-off library code, you do it every-single-fucking-time.

Every. Single. Fucking. Time.

Because you could doesn’t mean you should

Just like writing extremely verbose codes doesn’t mean better readability.

For example, I can name a function to be

func thisAddsTwoNumbersAndReturnItAsAString() -> String

Now I’m quoting SOLID principles, and claim this is the cleanest architecture:

I believe that I’ve created a modern & relatively simple pattern that can be used for literally anything.

And it is true! You can’t argue against self-explanatory function naming.

Can I call this architecture SNAKE and shove it down your throat now?

No, because you would save efforts and put it to more important aspects of production, like business logic or testing.

VIPER is doomed to fail from the start. I haven’t even gone into how SwiftUI removes the need for view model and intrinsic weakness of OOP.

Hello! Real World!

Let me give you a glimpse on what real world looks like.

Every couple months, some junior developer would come up proposing either MVVM/RxSwift/VIPER to use in production.

Not because they find the success of the said architecture during production, but because they suck at what they do and determine that it’s the awkward Apple SDK that fucked them.

And you know what usually happens when they try using it in production?

MVVM is just retarded. People would write 10 classes/protocols for a view model, which is nested and mixed with all the usual shit that made view controller a mess. Then they would claim it “solves” massive view controller problem not because code quality but because they read it on Internet.

RxSwift is just retarded. A colleague of mine just suggested me adding observer in a function that literally just returns a string from another string.

VIPER is beyond retarded. Most people can’t even get TODO app right. And once they get it right, the boilerplate is so retarded no one else would maintain it.

And most importantly, most of them can only write under the framework, never outside. For example Rx developer would turn everything including official SDK to Rx-based. They would tell you because Rx is so good. The truth is, Rx is not fitted for everything. And when they have to step outside Rx and actually build something, they suck.

Excuses excuses and excuses

I’ve just recently took over a RxSwift project. It has surprisingly small and clean view controllers, and it seems to greatly reduce the delegations required.

So RxSwift is not retarded?

Oh no, it is retarded. It has clean view controllers because it moves controller codes to views.

You can present view controller from view.

Let me say that again so there’s no confusion.

You can present view controller from view.

And you can do networking, navigation, have access to AppDelegate… etc, in a view.

And networking is custom-forked Rx-modified Alamofire, of which 90% usage can be easily done with URLSession. The rest 10% simply takes a little more efforts.

And view controller is wrapped as a member of view model.

You know I have to say this again so there’s no confusion.

And view controller is wrapped as a member of view model.

Oh RxSwift is retarded alright.

The take-away is this.

If you write shit in MVC, you write shit everywhere else

Stop preaching about retarded “clean” architecture if you can’t get the simplest thing right.
